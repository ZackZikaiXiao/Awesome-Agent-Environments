# Agent Environment: definition, taxonomy, trends, and construction patterns

> Working notes, updated in 2026-03.
> This document is intentionally more conceptual than the main README.

## 0. Why environments matter

In LLM agent research, model, data, and algorithm are only three of the main moving pieces. Environment is the fourth one, and it is often the most under-specified.

Why?

- It determines what the agent is allowed to observe and do.
- It determines whether success can be verified.
- It determines whether large-scale rollout is affordable.
- It determines whether the resulting capability transfers to the real world.

Classic reinforcement learning had Atari, Go, MuJoCo, and robotics simulators with crisp state transitions and clear rewards. Agent research is harder because the "world" is often a browser, an operating system, a code repository, or an evolving external service. Those worlds are expensive, slow, stateful, and often not fully reversible.

So the core environment question is not just "how do we benchmark agents?" It is:

> What world do we want agents to learn in, and how do we make that world executable, verifiable, scalable, and safe?

## 1. Working definition

This repository uses a deliberately broad definition:

> An agent environment is any system that accepts agent actions, maintains task-relevant state, returns observations and feedback, and eventually produces an outcome that can be evaluated.

This includes:

- real or self-hosted websites
- desktop and mobile operating systems
- dockerized code repositories
- app and API worlds
- multi-agent or social simulators
- synthetic environments generated by code or by world models
- runtime sandboxes used to host agent execution safely

### 1.1 Runtime boundary and capability surface

For agent systems, environment is not only a task world. It is also a runtime boundary.

That runtime boundary determines:

- what files an agent may read or write
- what networks it may call
- which tools and credentials are exposed
- what state is persistent across steps
- what is logged, traced, and reviewable later

This is why modern agent-environment work naturally spills into:

- sandboxing
- capability interfaces such as MCP
- orchestration and serving
- observability and debugging

## 2. Four orthogonal classification axes

README uses a single primary section per project for readability. This document uses orthogonal dimensions because that is how the field actually works.

### 2.1 Fidelity

From high realism to high controllability:

- `real`: live websites, real operating systems, real APIs
- `self-hosted`: cloned or re-hosted systems, usually resettable
- `llm-sim`: an LLM simulates the world or user
- `synthetic`: executable environments generated by code or rules
- `world-model`: neural or learned simulators

The central tradeoff is:

- higher fidelity gives better external validity
- higher controllability gives better scalability for training

You rarely get both for free.

### 2.2 Domain

Common environment domains:

- `web`
- `gui`
- `mobile`
- `coding`
- `terminal`
- `tool-use`
- `enterprise`
- `workflow`
- `embodied`
- `game`
- `social`
- `science`
- `multi-domain`

The domain matters because the bottleneck changes. Web needs DOM and visual grounding. OS and GUI need multi-app coordination. Coding needs executable verification. Tool-use needs compositional action chains. Embodied work needs sim-to-real transfer.

### 2.3 Purpose

- `eval`: environment is primarily for measurement
- `train`: environment is primarily for rollout and policy improvement
- `train+eval`: the same ecosystem supports both

This distinction matters because an evaluation benchmark can tolerate lower throughput, while a training gym needs:

- fast reset
- cheap parallel rollout
- reliable rewards
- stable reproduction

### 2.4 Verifiability

This is the most important axis for RL-style post-training.

- `fully-verifiable`: exact code-based or rule-based reward
- `partially-verifiable`: some state checks are executable, but not everything
- `llm-judge`: model-based scoring is required
- `human`: humans are still in the loop

The key idea is simple:

> Verifiability determines whether an environment can be used for scalable RLVR.

Code benchmarks are unusually strong because unit tests give hard feedback. Many web and GUI tasks are much harder because success may depend on fuzzy state, changing content, or long chains of latent intent.

## 3. Development patterns from 2023 to 2026

### 3.1 Benchmark saturation and successor churn

A recurring loop in agent environments is:

1. a benchmark is introduced
2. frontier models rapidly saturate it
3. a harder successor appears

Examples include:

- WebArena -> VisualWebArena -> WebChoreArena
- AndroidWorld -> MobileWorld
- OSWorld -> OSExpert
- SWE-bench -> Verified variants -> Live and Pro style successors

This suggests a structural issue:

> static benchmarks lag behind model progress

That is why "live" or continually refreshed benchmarks are becoming more important.

### 3.2 Shift from evaluation to training

2023 and 2024 were dominated by evaluation benchmarks.

2025 onward clearly shifts toward training environments:

- SWE-Gym
- R2E-Gym
- Debug-Gym
- RLVE
- synthetic environment generation pipelines

The underlying recognition is that pretrained LLMs already contain a lot of knowledge, but they still need environment-grounded experience.

### 3.3 Environment scaling as a new scaling axis

Recent work increasingly treats environment diversity as a scaling variable, not a side detail.

Representative works:

- EnvScaler
- ScaleEnv
- Agent World Model
- WebWorld
- GUI-GENESIS
- RLVE

The emerging hypothesis is:

> more diverse environments lead to better out-of-distribution agent generalization, much like more diverse data improves language-model generalization

### 3.4 Protocolization and observability become first-class

As environments become more tool- and system-heavy, the field is converging on two additional requirements:

- protocolized capability exposure
- standardized observability

Protocolized capability exposure means that environment access is no longer just a bundle of ad hoc Python functions. It is increasingly described through standard interfaces such as:

- MCP for tools and external systems
- OCI for runtime packaging
- WASI-style capability boundaries for portable execution

Standardized observability means that environment engineering now needs:

- traces for tool calls
- logs for policy and approval decisions
- low-level system visibility for sandbox debugging

Without these two layers, scaling agent environments becomes operationally fragile even when the benchmark design itself is sound.

### 3.4 Sandboxing becomes mandatory

As agents move from passive QA to active computer use, runtime isolation is no longer optional.

The field has moved from:

- local scripts and loose containers

toward:

- Docker
- gVisor
- Firecracker
- Kata
- managed sandbox platforms such as E2B
- cluster-level sandbox orchestration such as Kubernetes Agent Sandbox

The reason is straightforward:

- agents can mutate state
- agents can leak secrets
- agents can be injected
- long-running agent loops are hard to reason about without isolation and auditability

## 4. Canonical construction patterns

### 4.1 Real operating-system environments

Representative system: OSWorld

Pattern:

- run a full VM or real desktop stack
- expose screenshots and input events
- evaluate final state with scripts

Strengths:

- strong ecological validity
- no hidden API shortcuts

Weaknesses:

- slow rollout
- high maintenance cost
- UI drift breaks evaluators

### 4.2 Self-hosted web environments

Representative system: WebArena

Pattern:

- clone open-source web apps
- pre-populate realistic data
- host everything in resettable containers
- evaluate with backend state checks or page checks

Strengths:

- cheaper and more reproducible than the live web
- supports controlled experimentation

Weaknesses:

- distribution gap from the true web
- dynamic content and product changes are hard to mirror

### 4.3 Executable code and terminal environments

Representative systems:

- SWE-bench
- SWE-Gym
- R2E-Gym
- Debug-Gym
- Terminal-Bench

Pattern:

- start from a repository snapshot or a containerized task
- let the agent inspect, edit, and run tools
- evaluate with tests or exact state checks

Strengths:

- naturally verifiable
- ideal for RLVR and iterative training

Weaknesses:

- huge environment footprint
- dependency brittleness
- long-horizon sparse reward

### 4.4 Synthetic tool and app environments

Representative systems:

- EnvScaler
- ScaleEnv
- Agent World Model

Pattern:

- generate environment schema, tool interfaces, and stateful backends
- attach executable validators
- scale the number and diversity of worlds

Strengths:

- can produce many more environments than hand-authoring
- supports curriculum and diversity

Weaknesses:

- quality control is difficult
- generated environments can become unrealistic or shallow

### 4.5 World-model environments

Representative systems:

- WebWorld
- Dreamer 4
- VirtualEnv

Pattern:

- learn a simulator or latent transition model from trajectories
- train the policy inside the learned world

Strengths:

- potentially massive scaling benefits
- cheap imagination rollouts

Weaknesses:

- simulator bias
- uncertain long-horizon faithfulness
- domain transfer remains difficult

## 5. Core tensions

### 5.1 Fidelity vs controllability

Higher realism improves evaluation validity.
Higher controllability improves training throughput.

Most successful research stacks are layered:

1. train on controllable or synthetic worlds
2. evaluate on realistic worlds
3. study the transfer gap explicitly

### 5.2 Sparse reward vs scalable verification

Agent tasks are usually long horizon and terminally rewarded.

This creates a hard problem:

- many steps
- little feedback
- huge action space

Current approaches include:

- subgoal decomposition
- curriculum over difficulty
- hybrid verifiers
- self-play or self-generated tasks
- denser executable reward shaping

### 5.5 Environment engineering is now part of the product surface

In early agent work, environment design mostly lived inside papers and benchmark repositories.

That is no longer true.

Modern production agents expose environment engineering as user-visible product behavior:

- sandbox approvals
- network policies
- credential brokering
- workspace scoping
- auditable action logs

This means environment quality is no longer just a research concern. It directly affects:

- safety
- latency
- trust
- enterprise adoption

### 5.3 Benchmark contamination vs freshness

Public repositories, web pages, and issues can leak into model pretraining.
That makes a static benchmark less trustworthy over time.

Promising responses:

- continual or monthly refreshed benchmarks
- temporally held-out tasks
- unseen repositories and environments
- environment generation rather than benchmark memorization

### 5.4 Safety vs useful autonomy

Agent environments need enough capability to make tasks meaningful, but enough isolation to prevent harm.

This tension motivates:

- capability boundaries
- least-privilege tool exposure
- auditable action logs
- adversarial safety environments
- robust sandboxing by default

## 6. A practical framework for repository design

For an Awesome list, the hardest part is that real classification is multi-dimensional, while README contents must be tree-structured.

That is why this repository uses:

- one primary section per project in `README.md`
- tags for fidelity, domain, purpose, and verifiability
- a separate conceptual document, this file, for the orthogonal view

This avoids the common failure mode where a project like BrowserGym or GUI-GENESIS needs to be duplicated across many sections.

## 7. Open problems

The field still lacks clean answers to several important questions:

### 7.1 Unified cross-domain standards

We still do not have a universal OpenAI-Gym-for-agents standard spanning:

- web
- GUI
- mobile
- code
- tool-use
- enterprise workflows

### 7.2 Expanding verifiability

Many important agent tasks remain only partially verifiable.

The big challenge is:

> how do we turn more realistic tasks into tasks with reliable executable feedback?

### 7.3 Continually evolving environments

Static tasks are easy to overfit.
Future environments likely need:

- dynamic refresh
- adaptive difficulty
- contamination resistance
- anti-shortcut evaluation

### 7.4 Protocol and interface fragmentation

Even when two systems operate in the same domain, they often expose incompatible assumptions about:

- tool schemas
- authentication
- state persistence
- rollback behavior
- trace formats

This creates a hidden tax on benchmark reuse and agent transfer. A major future win would be better interop between:

- environment definitions
- runtime policies
- tool protocols
- observability pipelines

### 7.5 Multi-agent and socially grounded worlds

Most current agent environments are still single-agent.
Future work will need better abstractions for:

- asynchronous interaction
- shared state
- cooperation and competition
- organizational workflows

### 7.6 Formal safety boundaries

Sandboxing today is mostly engineering, not specification-first.
A more mature future stack would make it easier to declare:

- what an agent may read
- what it may write
- what external effects are allowed
- how violations are detected and audited
